## 🎯 架构师评审要点

请重点关注以下方面：

### 1. 模块拆分合理性
- 模块边界是否清晰
- 与其他模块的依赖关系是否合理
- 模块内聚性是否足够

### 2. 技术可行性
- 现有技术栈是否满足需求
- 技术方案是否可行
- 难点和风险点识别

### 3. 数据模型设计
- 字段定义是否合理
- 数据类型选择是否恰当
- 与其他模块的数据共享方案

### 4. 接口设计
- 模块间接口是否清晰
- 接口参数是否完整
- 异常情况处理

### 5. 性能考虑
- 大数据量处理方案
- 并发场景考虑
- 缓存策略

### 6. 扩展性
- 功能扩展预留
- 业务量增长应对

### 7. 与已完成模块的一致性
- 技术栈保持一致
- 编码规范统一
- 数据模型协调

## 🔄 评审通过条件

**重要**：只有当 `requirement_issue` 为空数组时，评审才算通过。

### 如果有 requirement_issue：
- 返回具体的修改建议
- 在 message 中明确告知产品需要修改的内容
- 流程将返回给产品进行修改，修改后再次评审

### 如果 requirement_issue 为空：
- 评审通过
- 在 message 中简要总结评审结论
- 流程进入下一阶段（整理风险和问题给客户）

## 📤 输出要求

完成评审后，请调用 `review_module_output` 工具输出结果，参数格式如下：

```json
{
  "requirement_issue": [
    // 只有发现问题才填写，如果模块已经完善则为空数组[]
    {
      "content": "针对模块提出的具体问题描述",
      "propose": "建议的解决方案"
    }
  ],
  "risks": [
    // 给客户的风险和建议方案
    {
      "content": "给客户提出的风险描述",
      "propose": "针对该风险的建议方案"
    }
  ],
  "unclear_points": [
    // 需求中不明确的问题
    {
      "content": "需求中不明确的问题描述",
      "propose": "针对该问题的建议方案"
    }
  ]
}
```

## ⚠️ 注意事项

- `requirement_issue` 为空 → 评审通过，进入下一阶段
- `requirement_issue` 有内容 → 需要产品修改，修改后再评审
- `risks` 和 `unclear_points` 始终需要填写（提给客户）
- 注意与已完成的模块保持技术一致性
