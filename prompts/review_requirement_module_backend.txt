## 🎯 后端工程师评审要点

请重点关注以下方面：

### 1. 接口设计评审
- 接口参数是否完整
- 接口返回值是否清晰
- 接口安全性考虑
- 接口幂等性设计

### 2. 数据处理逻辑
- 字段校验规则是否完整
- 数据计算逻辑是否正确
- 边界条件处理

### 3. 业务逻辑实现
- 状态流转是否完整
- 异常场景处理
- 事务控制方案

### 4. 外部依赖对接
- 第三方服务调用方案
- 超时和重试机制
- 降级策略

### 5. 数据库设计
- 表结构设计是否合理
- 索引设计考虑
- 数据迁移方案

### 6. 与已完成模块的一致性
- 接口规范统一
- 错误码规范一致
- 代码风格统一

## 🔄 评审通过条件

**重要**：只有当 `requirement_issue` 为空数组时，评审才算通过。

### 如果有 requirement_issue：
- 返回具体的修改建议
- 在 message 中明确告知产品需要修改的内容
- 流程将返回给产品进行修改，修改后再次评审

### 如果 requirement_issue 为空：
- 评审通过
- 在 message 中简要总结评审结论
- 流程进入下一阶段（整理风险和问题给客户）

## 📤 输出要求

完成评审后，请调用 `review_module_output` 工具输出结果，参数格式如下：

```json
{
  "requirement_issue": [
    // 只有发现问题才填写，如果模块已经完善则为空数组[]
    {
      "content": "针对模块提出的具体问题描述",
      "propose": "建议的解决方案"
    }
  ],
  "risks": [
    // 给客户的风险和建议方案
    {
      "content": "给客户提出的风险描述",
      "propose": "针对该风险的建议方案"
    }
  ],
  "unclear_points": [
    // 需求中不明确的问题
    {
      "content": "需求中不明确的问题描述",
      "propose": "针对该问题的建议方案"
    }
  ]
}
```

## ⚠️ 注意事项

- `requirement_issue` 为空 → 评审通过，进入下一阶段
- `requirement_issue` 有内容 → 需要产品修改，修改后再评审
- `risks` 和 `unclear_points` 始终需要填写（提给客户）
- 注意与已完成的模块保持后端实现一致性
